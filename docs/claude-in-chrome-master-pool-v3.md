> **문서 성격**: Claude in Chrome으로 공략할 "API 적대 플랫폼" 마스터 리스트 + 전략 근거 + 운용 설계
> **작성일**: 2026.02.19
> **버전**: v3.0 (최종 확정판)
> **전제**: 피지컬 로컬 비즈니스 기반 (호야당/연화스튜디오/동선스튜디오)
> **자체 평가**: 95/100 (나머지 5점은 첫 실행 완료 시 자동 충족)

## 0. 이 백서의 취지

### 왜 이 문서가 필요한가

Claude in Chrome은 "API가 막힌 플랫폼의 GUI 클릭질을 비전 AI로 대체하는 도구"다.

**원칙: API가 열려있으면 Claude in Chrome을 쓸 이유가 없다.** GitHub, Discord, X/Twitter는 API로 자동화가 된다. 이 도구의 존재 이유는 오직 하나 — API를 알러지 반응처럼 차단하고, 로봇을 적대시하고, 사람의 수작업 클릭질을 강요하는 "지랄맞은 플랫폼"을 뚫는 것이다.

### 역발상 전략

```
기존 접근: "API 있는 쉬운 데부터" → 이미 해결된 문제를 또 푼다
이 백서:   "API 막힌 지랄맞은 데부터" → 그동안 안 됐던 걸 뚫는다
```

가장 막혀있던 놈부터 뚫는 게 ROI가 가장 높다. 쉬운 건 이미 되고 있으니까.

### 이 백서의 위치

이 문서는 전산 베이스 시스템의 **설계도**다. 플랜이 완벽해야 실행한다. 설계도 없이 뛰어드는 게 병신이지, 설계를 꼼꼼히 하는 게 느린 게 아니다.

### 오퍼레이션 시퀀스

```
[지금] Phase 0: 플랫폼 선별 + 전략 설계 + 시스템 아키텍처 ← 이 백서
  ↓
[다음] Phase 1: 시스템 구축 + 첫 실행 (Record Workflow + 프롬프트 검증)
  ↓
[이후] Phase 2: 트래픽 발생 + 데이터 수집
  ↓
[그때] Phase 3: 성과 측정 + KPI 설계 + 최적화
```

KPI는 Phase 3에서 한다. 트래픽 없는데 KPI 세우는 건 빈 엑셀에 그래프 그리는 거다. 지금은 Phase 0이다.

### 이 시스템의 주체

내 가게, 내 지인 가게에 내가 직접 설치하는 시스템이다. 대행 서비스가 아니다. 비교 대상이 없다. 경쟁 분석은 서비스화 단계에서 필요하면 그때 한다.

## 1. 플랫폼 선별 기준

### 1.1 3조건 동시 충족 필터

```
조건 1: API 적대도 A급 이상 (자동화가 막혀있다)
조건 2: 돈에 직결 (매출/수익에 직접 연결된다)
조건 3: 피지컬 로컬 비즈니스에 필요 (호야당/연화/스튜디오에 실제로 쓴다)
```

3개 중 하나라도 빠지면 이 백서에 안 넣는다.

### 1.2 API 적대도 등급

| 등급 | 정의 | 판정 |
|---|---|---|
| **S급** | API 완전 폐쇄 + 봇 탐지 공격적 + 자동화 시도 시 계정 제재 | 최우선 공략 |
| **A급** | API 없음 또는 폐지 + GUI만 허용 + 외부 도구 차단 | 공략 대상 |
| **B급** | API 있지만 GUI 전용 기능 많음 + 대시보드 종속 | 조건부 포함 |
| **C급** | API 열려있음 + 자동화 가능 | 이 백서 범위 밖 |

### 1.3 실제 비즈니스 매칭

| 비즈니스 | 성격 | 핵심 플랫폼 |
|---|---|---|
| 호야당 (빵집) | 피지컬 로컬 매장 | 플레이스 → 인스타 → 당근 |
| 연화 스튜디오 (탱고) | 피지컬 로컬 + 프리미엄 | 플레이스 → 인스타 → 유튜브 |
| 동선 스튜디오 (영상) | 피지컬 로컬 + 서비스 | 플레이스 → 인스타 → 유튜브 |
| dtslib 노드들 (폰가게/부동산 등) | 피지컬 로컬 각종 | 플레이스 → 당근 → 블로그 |
| parksy.kr (콘텐츠) | 디지털 퍼블리싱 | 블로그 → 유튜브 → 티스토리 |

## 2. 최종 마스터 풀: 6개

### K1. 네이버 플레이스

| 항목 | 내용 |
|---|---|
| **API 적대도** | A급. 외부 API 없음. GUI만. 사업자등록증 필수. 모든 관리가 스마트플레이스 대시보드 수작업 |
| **수익 직결도** | 직접 매출. "안양 빵집", "강남 탱고" 검색 = 네이버 지도 플레이스가 먼저 뜸 = 매장 방문 = 매출 |
| **왜 K1인가** | 한국에서 로컬 장사하면 플레이스가 생명줄이다. 블로그 글보다 지도 플레이스가 검색 결과 상단에 뜬다. 매장 전화, 길찾기, 예약 — 전부 플레이스에서 시작. 호야당, 연화, 동선, 폰가게, 부동산 — 피지컬 노드 전부 해당 |
| **Claude in Chrome 작업** | 업체 정보 업데이트(영업시간/메뉴/사진), 소식/이벤트 등록, 리뷰 답글 작성, 예약 설정 관리 |
| **클릭질 고통도** | 높음. 소식 하나 올리려면 로그인 → 소식관리 → 작성 → 이미지 업로드 → 발행. 매장 여러 개면 계정 전환까지. 주 2-3회 필수 |

### K2. Instagram

| 항목 | 내용 |
|---|---|
| **API 적대도** | S급. API 사실상 폐쇄. 봇 탐지 가장 공격적. 클릭 패턴/IP/디바이스 핑거프린트 전부 감시. 자동화 시도 시 계정 잠금 |
| **수익 직결도** | 직접 매출. 로컬 장사에서 네이버 다음 유입 채널. 빵 사진 → "가보고 싶다" → 방문. 탱고 릴스 → "배우고 싶다" → 문의 |
| **왜 K2인가** | API 적대도 최강이라 Claude in Chrome ROI가 가장 높은 플랫폼. 비전 기반이라 인스타가 DOM을 아무리 바꿔도 적응. 피지컬 비즈니스에서 인스타 없으면 존재감 없음. 글로벌 확장도 같은 계정에서 영문 해시태그만 추가하면 됨 |
| **Claude in Chrome 작업** | 게시물/릴스 업로드, 캡션 작성, 해시태그 30개 입력, 스토리 게시 |
| **클릭질 고통도** | 극높음. 캐러셀 하나에 이미지 선택 → 크롭 → 필터 → 캡션 → 해시태그 30개 → 위치태그 → 게시. 매일 하면 미친다 |

### K3. 네이버 블로그

| 항목 | 내용 |
|---|---|
| **API 적대도** | A급. API 없음. 자체 WYSIWYG 에디터 강요. 마크다운 불가. 외부 에디터 붙여넣기 포맷 깨짐 |
| **수익 직결도** | 네이버 검색 1위. 블로그 → 플레이스 → 매장 방문 퍼널. 블로그 리뷰 많으면 플레이스 노출 상승 |
| **왜 K3인가** | 네이버 생태계에서 블로그와 플레이스는 연동. 블로그 리뷰가 많을수록 플레이스 순위가 올라간다. K1의 보조 엔진. parksy.kr 콘텐츠를 네이버 생태계로 유입시키는 채널 |
| **Claude in Chrome 작업** | 네이버 에디터에 본문 입력(줄간격/구분선/이미지), 태그, 카테고리, 발행 |
| **클릭질 고통도** | 높음. 네이버 에디터가 특히 짜증. 이미지 하나에 업로드 → 크기조절 → 정렬. 글 하나 15-20분 |

### K4. 당근마켓

| 항목 | 내용 |
|---|---|
| **API 적대도** | A급. 외부 API 전무. 앱/웹 GUI만. 자동화 도구 자체가 존재하지 않는 폐쇄 생태계 |
| **수익 직결도** | 직접 매출. 반경 6km 동네 기반. 비즈프로필 = 무료 매장 홍보. 동네홍보 = 직접 유입 |
| **왜 K4인가** | 호야당 같은 동네 빵집에 가장 직접적. "우리 동네 빵집" 느낌 동네홍보 → 반경 6km 주민 직접 노출. 당근은 "동네 신뢰" 기반이라 전환율이 높다. 폰가게도 당근이 핵심 |
| **Claude in Chrome 작업** | 비즈프로필 소식, 동네홍보 글, 상품 등록/수정 |
| **클릭질 고통도** | 중간. 단건은 간단하지만 비즈프로필+동네홍보+상품 주기적 관리하면 반복 노동 |

### K5. YouTube

| 항목 | 내용 |
|---|---|
| **API 적대도** | B급. API로 업로드는 가능하나 Studio GUI 전용 기능 다수 — 썸네일 A/B, 분석, 커뮤니티 탭, 자막, 카드/엔드스크린, 수익화 설정 |
| **수익 직결도** | 광고 수익 직접 발생. 장기 브랜딩 엔진. 호야당 제빵 과정, 연화 탱고 레슨, 추모영상 포트폴리오 |
| **왜 K5(Phase 2)인가** | 영상은 촬영→편집→업로드 파이프라인이 필요해서 텍스트+사진 플랫폼(K1-K4)보다 진입 시간이 길다. 하지만 장기적으로 가장 강력한 브랜딩+수익 채널 |
| **Claude in Chrome 작업** | 영상 업로드 + 제목/설명/태그, 썸네일, 커뮤니티 탭, 자막, 카드/엔드스크린 |
| **클릭질 고통도** | 높음. 건당 제목/설명/태그/썸네일/카드/엔드스크린/공개설정까지 15-20분 |

### K6. 티스토리

| 항목 | 내용 |
|---|---|

## 7. Claude in Chrome 작동 원리

### 왜 기존 자동화와 다른가

| | 셀레니움/태스커 | Claude in Chrome |
|---|---|---|
| 인식 방식 | DOM 셀렉터 (class명, id) | 스크린샷 + Accessibility Tree 하이브리드 |
| DOM 변경 시 | 깨짐. 매번 셀렉터 수정 | 비전 기반이라 적응. 돋보기=검색으로 인식 |
| 콘텐츠 이해 | 없음. 고정 텍스트만 입력 | 원본 읽고 플랫폼별 톤 변환 |
| 결과 | 클릭 매크로 | 맞춤 변환 + 업로드 한 번에 |

## 10. 운용 원칙

1. **원본은 내 도메인에** — parksy.kr = SSOT. 플랫폼은 배포 채널.
2. **변환은 AI, 판단은 나** — 톤 변환은 Claude. 발행 승인은 나.
3. **자동화는 퍼블리싱만** — 인게이지먼트 자동화 절대 금지.
4. **지랄맞은 놈부터 뚫는다** — API 적대도 가장 높은 플랫폼 먼저.
5. **글로벌은 로컬의 확장** — 별도 플랫폼 추가 안 한다.
6. **설계 먼저, 실행은 그 다음** — 전산 베이스 시스템이다. 플랜이 완벽해야 실행한다.
