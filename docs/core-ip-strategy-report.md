## 4. Constraint-to-Moat Framework (제약→해자 전환)

### 정의

```
"제약 조건을 경쟁 우위로 전환하는 프레임워크"

핵심 개념:
├── 제약 = 핸디캡이 아닌 기회
├── 남들이 안 하는 이유 = 내가 독점하는 이유
├── 불편함을 시스템으로 해결 = 해자
└── 시간이 갈수록 격차 벌어짐
```

### 전환 사례

| 제약 | 일반적 반응 | DONGSEON 전환 | 결과 |
|------|------------|---------------|------|
| 12시간 노동 | 포기 | 자동화 필수화 | 남들은 자동화 안 함 |
| PC 없음 | 개발 불가 | 핸드폰 개발 | 남들은 핸드폰 개발 못 함 |
| 시간 없음 | 편집 포기 | 무편집 워크플로우 | 남들은 편집에 시간 씀 |
| 독학 | 한계 | 가르칠 사람 = 나뿐 | 교육 독점 |
| 자본 없음 | 사업 불가 | 0원 인프라 | 마진 100% |

### 프레임워크 구조

```
┌────────────────────────────────────────────────────────────┐
│           Constraint-to-Moat Framework                     │
├────────────────────────────────────────────────────────────┤
│                                                            │
│   Step 1: 제약 인식                                        │
│   └── "나는 ___를 못한다/없다"                             │
│                                                            │
│   Step 2: 역발상                                           │
│   └── "그러면 ___를 안 해도 되게 만들자"                   │
│                                                            │
│   Step 3: 시스템화                                         │
│   └── "___없이 ___하는 방법을 체계화"                      │
│                                                            │
│   Step 4: 해자 형성                                        │
│   └── "남들은 ___가 있어서 이 방법 안 씀"                  │
│   └── "나만 이 방법 쓰니까 독점"                           │
│                                                            │
│   Step 5: 시간 경과                                        │
│   └── "노하우 축적 → 격차 확대 → 진입장벽"                │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 평가

| 항목 | 점수 | 근거 |
|------|------|------|
| 독창성 | 10/10 | 체계화된 프레임워크 없음 |
| 실용성 | 9/10 | 적용에 사고 전환 필요 |
